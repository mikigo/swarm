# Requirements Analysis & Architecture Design Skill

## 概述

本 Skill 定义了从需求分析到文档产出的完整工作流程，适用于任何需要设计分布式系统或客户端/服务端架构的项目。

## 适用场景

- 新项目从 0 开始
- 需要设计分布式系统
- 需要定义客户端/服务端架构
- 需要与用户逐项确认需求

## 使用方法

在 OpenCode 中，通过 `skill` 工具加载此 skill：
```
加载后，按照工作流程逐步与用户确认需求。
```

## 核心原则

### 1. 逐个确认原则

每个需求问题必须单独确认：
- 先给出方案对比（包含优点/缺点）
- 给出分析和建议
- 让用户选择

**禁止**：一次性列出多个问题让用户确认

### 2. 分析先行原则

每个问题必须包含：
- **方案对比表**：至少 2-3 个方案对比
- **我的分析**：推荐某个方案及理由
- **最终问题**：让用户确认

格式：
```
### 方案对比

| 方案 | 优点 | 缺点 |
|------|------|------|
| A    | xxx  | xxx  |
| B    | xxx  | xxx  |

### 我的分析

推荐 X，原因：
1. ...
2. ...
3. ...

**选 A、B 还是 C？**
```

### 3. 文档同步原则

每确认一个关键决策，立即更新文档中的相关章节，避免最后集中修改。

## 工作流程

### Phase 1: 需求调研

1. **了解现状**
   - 是否有现有代码/项目
   - 是否有参考竞品
   - 项目的技术栈偏好

2. **核心需求确认**
   - 服务端/客户端架构
   - 通信方式（HTTP/WebSocket/gRPC）
   - 部署模式（守护进程/按需）
   - 功能范围

### Phase 2: 逐项确认

按照以下顺序逐个确认（可调整）：

| # | 主题 | 示例问题 |
|---|------|----------|
| 1 | 通信协议 | HTTP / WebSocket / gRPC |
| 2 | 网络模式 | 直连 / 反向连接 |
| 3 | 分发策略 | 动态按需 / 批量 / 轮询 |
| 4 | 数据传输 | 文件路径 / Item 对象 |
| 5 | 结果回传 | HTTP 上传 / WebSocket 推送 |
| 6 | 外部依赖 | Allure、Git 等 |
| 7 | 客户端管理 | 服务端推送 / 客户端拉取 |
| 8 | 代码同步 | Git clone / 共享存储 / 本地配置 |
| 9 | 虚拟环境 | 独立 / 复用 / 不需要 |
| 10 | 任务创建 | CLI / API / Web UI |
| 11 | 认证方式 | 无 / Token / API Key |
| 12 | 报告访问 | 静态托管 / 内嵌查看器 |
| 13 | 安装方式 | pip / Docker / 二进制 |
| 14 | 参数透传 | 前缀区分 / 环境变量 / 配置 |
| 15 | 任务状态 | 如何定义成功/失败 |
| 16 | 心跳超时 | 超时后如何处理 |
| 17 | CLI 设计 | 子命令 / 独立入口 |

### Phase 3: 补充讨论

确认核心需求后，补充讨论：
- 并发控制
- 超时处理
- 重试机制
- 日志保留
- 监控告警
- 其他非功能需求

### Phase 4: 文档产出

产出文档：
1. **需求规格说明书** (requirements.md)
   - 项目概述
   - 功能需求
   - 数据模型
   - 执行流程
   - 验收标准

2. **架构设计文档** (architecture.md)
   - 系统架构
   - 通信设计
   - 调度设计
   - 数据存储
   - 异常处理

### Phase 5: 歧义检查

完成文档后，检查是否有：
- 表述不明确的地方
- 可能产生歧义的地方
- 实现时可能遇到问题的地方

整理成问题清单，逐个与用户确认。

## 方案模板

以下是在分布式系统中常见的技术方案选择：

### 通信协议

| 方案 | 优点 | 缺点 |
|------|------|------|
| HTTP + 轮询 | 简单，防火墙友好 | 实时性差 |
| WebSocket | 实时双向，效率高 | 需要维护连接 |
| gRPC | 高效，类型安全 | 生态较新 |

### 分发策略

| 方案 | 优点 | 缺点 |
|------|------|------|
| 动态按需 | 智能调度，失败快速重试 | 实现较复杂 |
| 批量分发 | 简单，减少网络请求 | 负载不均 |
| 轮询拉取 | 简单，负载均衡 | 无法动态调整 |

### 代码同步

| 方案 | 优点 | 缺点 |
|------|------|------|
| Git clone | 简单可靠，支持版本 | 每次需要网络 |
| 共享存储 | 无需网络传输 | 需要 NFS 等 |
| 本地配置 | 速度快 | 客户端需要预配置 |

### 虚拟环境

| 方案 | 优点 | 缺点 |
|------|------|------|
| 任务独立 | 隔离好，无污染 | 每次创建耗时 |
| 环境复用 | 速度快 | 可能有依赖冲突 |
| 无虚拟环境 | 最简单 | 环境污染 |

---

## 输出要求

文档必须包含：
1. 清晰的目录结构
2. 完整的 API 设计
3. 数据模型示例（JSON）
4. 执行流程图
5. 与竞品对比（如有）
6. 技术选型说明

---

## 注意事项

1. **不要假设**：任何不确定的地方都要问用户
2. **不要跳过**：即使是最简单的问题，也要确认
3. **不要合并**：不要一次性问多个问题
4. **记录理由**：每个决策都要记录理由，方便后续查阅
5. **保持迭代**：文档是活的，确认后立即更新

## 示例：常见问题确认话术

### 问题 1：通信协议

```
### 方案对比

| 方案 | 优点 | 缺点 |
|------|------|------|
| A. WebSocket | 实时双向，效率高 | 需要维护连接 |
| B. HTTP + 轮询 | 简单，防火墙友好 | 实时性差 |
| C. gRPC | 高效，类型安全 | 生态较新 |

### 我的分析

推荐 A，原因：
1. 实时双向通信是核心需求
2. WebSocket 在 FastAPI 中原生支持
3. 动态调度需要双向通信

**选 A、B 还是 C？**
```

### 问题 2：任务状态

```
### 方案对比

| 方案 | 描述 |
|------|------|
| A. 用例失败=任务失败 | 任何用例失败，任务标记为失败 |
| B. 任务只关心是否执行完 | 用例可以失败，只要执行完就算成功 |

### 我的分析

推荐 B，原因：
1. 更灵活，测试结果的判定应该由用户决定
2. 符合"任务执行"的概念，执行完成即成功
3. 便于后续扩展支持自定义失败判定

**选 A 还是 B？**
```
