# 需求分析与架构设计 Skill

## 概述

本 Skill 定义了分布式测试框架（Swarm）从需求分析到文档产出的完整工作流程。

## 适用场景

- 新项目从 0 开始
- 需要设计分布式系统
- 需要定义客户端/服务端架构

## 核心原则

### 1. 逐个确认原则

每个需求问题必须单独确认：
- 先给出方案对比（包含优点/缺点）
- 给出分析和建议
- 让用户选择

**禁止**：一次性列出多个问题让用户确认

### 2. 分析先行原则

每个问题必须包含：
- **方案对比表**：至少 2-3 个方案对比
- **我的分析**：推荐某个方案及理由
- **最终问题**：让用户确认

格式：
```
### 方案对比

| 方案 | 优点 | 缺点 |
|------|------|------|
| A    | xxx  | xxx  |
| B    | xxx  | xxx  |

### 我的分析

推荐 X，原因：1... 2... 3...

**选 A、B 还是 C？**
```

### 3. 文档同步原则

每确认一个关键决策，立即更新文档中的相关章节，避免最后集中修改。

## 工作流程

### Phase 1: 需求调研

1. **了解现状**
   - 是否有现有代码/项目
   - 是否有参考竞品
   
2. **核心需求确认**
   - 服务端/客户端架构
   - 通信方式
   - 部署模式
   - 功能范围

### Phase 2: 逐项确认

按照以下顺序逐个确认（可调整）：

| 顺序 | 主题 | 示例问题 |
|------|------|----------|
| 1 | 通信协议 | HTTP / WebSocket / gRPC |
| 2 | 网络模式 | 直连 / 反向连接 |
| 3 | 分发策略 | 动态按需 / 批量 / 轮询 |
| 4 | 数据传输 | 文件路径 / Item 对象 |
| 5 | 结果回传 | HTTP 上传 / WebSocket 推送 |
| 6 | 外部依赖 | Allure、Git 等 |
| 7 | 客户端管理 | 服务端推送 / 客户端拉取 |
| 8 | 代码同步 | Git clone / 共享存储 / 本地配置 |
| 9 | 虚拟环境 | 独立 / 复用 / 不需要 |
| 10 | 任务创建 | CLI / API / Web UI |
| 11 | 认证方式 | 无 / Token / API Key |
| 12 | 报告访问 | 静态托管 / 内嵌查看器 |
| 13 | 安装方式 | pip / Docker / 二进制 |
| 14 | 参数透传 | 前缀区分 / 环境变量 / 配置 |

### Phase 3: 补充讨论

确认核心需求后，补充讨论：
- 并发控制
- 超时处理
- 重试机制
- 日志保留
- 监控告警
- 其他非功能需求

### Phase 4: 文档产出

产出文档：
1. **需求规格说明书** (requirements.md)
   - 项目概述
   - 功能需求
   - 数据模型
   - 执行流程
   - 验收标准

2. **架构设计文档** (architecture.md)
   - 系统架构
   - 通信设计
   - 调度设计
   - 数据存储
   - 异常处理

### Phase 5: 歧义检查

完成文档后，检查是否有：
- 表述不明确的地方
- 可能产生歧义的地方
- 实现时可能遇到问题的地方

整理成问题清单，逐个与用户确认。

## 关键设计决策记录

| 决策点 | 选择 | 理由 |
|--------|------|------|
| 通信协议 | WebSocket + HTTP | 实时双向，简洁高效 |
| 分发策略 | 动态按需 | 类似 pytest-xdist，智能调度 |
| 用例粒度 | 文件级 | 避免重复收集，调试友好 |
| 代码同步 | 每次 clone | 简单可靠，支持分支/Tag |
| 虚拟环境 | 任务独立 | 隔离性好，避免污染 |
| 报告生成 | 服务端汇总 | 统一入口，便于管理 |
| 部署模式 | 单包安装 | 简化部署，内网友好 |
| 客户端验证 | 无 | 内网部署，安全由网络层保证 |

## 输出要求

文档必须包含：
1. 清晰的目录结构
2. 完整的 API 设计
3. 数据模型示例（JSON）
4. 执行流程图
5. 与竞品对比（如有）
6. 技术选型说明

## 注意事项

1. **不要假设**：任何不确定的地方都要问用户
2. **不要跳过**：即使是最简单的问题，也要确认
3. **不要合并**：不要一次性问多个问题
4. **记录理由**：每个决策都要记录理由，方便后续查阅
5. **保持迭代**：文档是活的，确认后立即更新
